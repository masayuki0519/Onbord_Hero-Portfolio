# Game Design

## Project Goal

オンボードヒーロー is a portrait mobile idle RPG. The final goal is clearing the final stage. Players repeat combat, treasure chest opening, equipment collection, class growth, Job growth, and rune growth to increase combat power.

## Core Loop

1. Defeat normal enemies.
2. Collect treasure chests.
3. Open chests for equipment or class shards.
4. Upgrade level, skills, runes, equipment, and Jobs.
5. Challenge bosses at floor 10.
6. Clear bosses to unlock the next stage.

## Combat

- Normal enemies are fought automatically.
- Tapping anywhere on the battle area performs a manual attack.
- Critical hits use a special damage popup.
- Boss challenge starts only from floor 10.
- Boss challenge restores player HP to max and resets boss HP every attempt.
- If the player is defeated, boss combat is cancelled and enemy HP is reset.
- A boss fight opens with a short (900ms) entrance beat: all damage
  (player and enemy) is paused, the boss sprite scales in with a light
  flash, and the boss's name is announced via a toast and the boss ribbon
  (`⚔ ボス戦: <name>`). This exists so a boss fight registers as an event
  even when the boss dies in only 1-2 hits; it is not meant to be a hard
  gate on boss difficulty. The enemy HP bar also gets a distinct
  thicker/gold style during boss fights.

## Combat Power

The player-facing power metric is Combat Power, not Equipment Power. Combat Power is a combined score from:

- Attack
- HP
- Defense
- Critical rate
- Critical damage
- Attack speed
- Sustain stats such as regen, lifesteal, and evasion
- Equipment power

Boss recommendations use Combat Power. Recommended values should be tuned against actual clear feel, not equipment power alone.

## Stages And Portal

- Stages are grouped by difficulty, chapter, and floor.
- The battle screen has a Portal button opposite the Boss Challenge button.
- Portal opens stage selection.
- Locked stages are disabled and grayed out.
- `maxUnlockedStage` is the unlock source for stage selection.

Difficulty names:

- ノーマル
- ハード
- ヘル
- ナイトメア
- 深淵
- 天国
- 天界

## Mini Window (Overlay)

A native Android "draw over other apps" overlay widget shows a compact
status readout (title / stage / level / HP / coins / chest count) while
the user is outside the app, and can be double-tapped to relaunch the app.

- Native implementation: `MiniWindowController.java` (single static
  overlay view — see `docs/AI_RULES.md` for the singleton note).
- JS side (`updateMiniWindow()` in app.js) pushes a structured snapshot
  (coins/XP + their per-minute rates, plus the static title/stage/HP/chest
  text) to the native view via the Capacitor overlay plugin. See "Native
  live-ish projection" below for how this snapshot is used.
- Double-tapping the overlay opens the app and hides the overlay
  (`MiniWindowController.openApp()` calls `hide()`); tapping the "×"
  button hides it without opening the app.
- The JS-side refresh is driven by a `setInterval` (not
  `requestAnimationFrame`), because rAF callbacks stop firing entirely
  once the WebView page is hidden/backgrounded — exactly when the overlay
  is meant to be useful. This JS-side interval is *not* what keeps the
  overlay's coins/XP moving while backgrounded any more (see below) — it's
  now just how often JS hands the native side a fresh, accurate snapshot
  while the app *is* actually running.
- **Spec: the mini window's coins/XP are a native-side projection, not a
  live simulation.** The overlay never runs any game logic and never
  touches `state.coins`/`state.xp` — it only extrapolates forward from the
  last real snapshot JS handed it, using the same per-minute rates
  `applyOfflineProgress()` uses for the real catch-up. HP/stage/level/
  chest-count are not projected (frozen at the last real snapshot) since
  the offline-progress model never simulated those either. A definite,
  correct catch-up still happens the moment the player returns
  (`applyOfflineProgress()`) — the projection is a display nicety on top
  of that guarantee, not a replacement for it.

### SE Fully Muted While Backgrounded (2026-07-08, Priority Round 25)

The game loop keeps running for real behind the mini window (see above),
so combat/loot events keep firing sound-effect calls the entire time it's
up. `pauseBgmForBackground()`/`resumeBgmFromBackground()` (toggled from
the `visibilitychange` handler) only ever silenced BGM — every other SE
(attack/hit/coin/level/drop/chest/open/boss/button, all routed through
`playSound()`) kept playing audibly while the phone showed only the mini
window. Fixed with a single `audioMutedForBackground` flag set/cleared by
those same two functions and checked at the top of `playSound()` — one
choke point for every non-BGM sound, no per-call-site changes needed.

### Native live-ish projection (2026-07-06, Priority Round 6)

Priority Round 5's `@capacitor/app` `pause`/`resume` events fixed
*resume-time* catch-up reliability, but device feedback afterward made
clear that wasn't the actual ask: an idle game needs to feel like it's
progressing while the mini window is up, not just deliver a lump sum
whenever the player happens to come back. Two prior attempts at a
JS-`setInterval`-driven live display (2026-07-05's original refresh,
2026-07-06's `projectedOfflineCoins()`) both failed on-device for the same
reason: Chromium throttles/freezes the WebView's own JS timers once
backgrounded (see "Why: investigation findings" below), and that's true
no matter what the interval callback computes.

The fix moves the *projection itself* out of JS and into native Java,
sidestepping that specific failure mode entirely:
- `MiniWindowController` keeps a small snapshot (`baseCoins`,
  `baseCoinsPerMinute`, `baseXp`, `baseXpPerMinute`, `baseXpNeeded`, plus
  the static title/stage/HP/chest text) and a `SystemClock.elapsedRealtime()`
  timestamp of when it was captured.
- `updateMiniWindow()` (JS) sends this snapshot on every real update while
  the app is running (via the Capacitor plugin bridge,
  `OverlayPermissionPlugin.updateMiniWindow` → `MiniWindowController.updateProjection`).
  `coinsPerMinute` is `damage() * 0.35` and `xpPerMinute` is a flat `0.5` —
  copied directly from `applyOfflineProgress()`'s real formula, not a
  separate guess, so the projection can't drift from what the real
  catch-up will eventually grant.
- A `Handler` tied to the app's main `Looper` (**not** a foreground
  Service — no new permission, no persistent notification) re-renders the
  overlay text from that snapshot every 20 seconds
  (`TICK_INTERVAL_MS`), for as long as the overlay `View` exists. This
  runs entirely in the host app's own Java code, independent of the
  WebView's JS engine, so it isn't subject to Chromium's background-page
  timer throttling — a `Handler.postDelayed` loop on the main thread of a
  still-alive process is a fundamentally different mechanism than a JS
  `setInterval` inside a hidden WebView page.
  Capped at the same 240-minute horizon `applyOfflineProgress()` uses, so
  the projected number can't run away to an absurd value if the app is
  never reopened.
- The projected XP is capped at `xpNeeded - 1` (never rolls over into a
  simulated level-up) — the native side deliberately does *not* duplicate
  the leveling formula (`state.level * 7` thresholds, `while` loop in
  `defeatEnemy()`), to avoid a second copy of that game rule to keep in
  sync forever. A real level-up still only ever happens via
  `applyOfflineProgress()`/`defeatEnemy()` in JS.
- Ticking starts in `MiniWindowController.show()` and stops in `hide()`
  (`stopTicking()`), so there's no perpetual timer once the overlay is
  actually closed.
- No new Android permission and no persistent notification — this is
  explicitly *not* the foreground-Service approach evaluated and rejected
  in Priority Round 3 (see below); it's a lighter-weight mechanism with a
  different risk profile (main-thread `Handler`, not a Service component),
  though it inherits the same ultimate limit: if Android kills the whole
  app process (e.g. under memory pressure), the tick stops like everything
  else in the process, and the next launch's cold-start
  `applyOfflineProgress()` call still settles the correct amount.
- Verified: full Gradle build (`compileDebugJavaWithJavac` and
  `assembleDebug`) succeeded in this environment (an Android
  SDK/JBR toolchain turned out to be available here, unlike in earlier
  sessions) — this is a real compiled-and-packaged verification, not just
  a manual code read. **Not yet verified on an actual device/emulator**
  (none was connected in this environment) — needs on-device confirmation
  that the overlay text visibly ticks up while backgrounded.

### Why: investigation findings (2026-07-06)

A `setInterval`-based approach was tried twice and failed on-device both
times (device-confirmed not working, both 2026-07-05's original
mini-window refresh interval and 2026-07-06's coin-projection attempt
below). Root-caused by reading Capacitor's own Android source
(`node_modules/@capacitor/android`), not just guessing:

- `BridgeActivity.onPause()` → `Bridge.onPause()` →
  `MockCordovaWebViewImpl.handlePause(keepRunning)`. `keepRunning` comes
  from `shouldKeepRunning()` = `preferences.getBoolean("KeepRunning", true)`
  — this project has no `KeepRunning` preference set (checked
  `android/app/src/main/res/xml/config.xml`), so it defaults to `true`.
  The method only calls `webView.pauseTimers()` when `keepRunning` is
  `false`, so **Capacitor's own Java-level lifecycle handling does not
  pause JS timers for this app** — `setInterval` should, in principle,
  keep running at the Capacitor/Android-API level.
- However, Android's WebView is Chromium-based, and Chromium applies its
  *own* internal, engine-level background-page power-saving policy
  (timer throttling that intensifies over time, and eventual page
  "freezing" for pages hidden more than a few minutes) **independently of
  the Java-level `WebView.pauseTimers()` API** — this is a Blink/V8
  scheduler decision, not something the embedding Android app's Java code
  controls or can opt out of via Capacitor config. This most likely
  explains why `document.hidden`/`visibilitychange` fire reliably (that's
  a page-visibility *signal*, sent regardless) while `setInterval`
  callbacks stop being observed in practice once the app has been
  backgrounded for more than a short window — the standing assumption
  ("`setInterval` keeps pushing, just throttled") from the original
  2026-07-05 mini-window fix undersold how aggressive this can get for
  realistic "minimized for a few minutes" usage, as opposed to a few
  seconds.
- Net effect: **the DOM update call itself (`updateMiniWindow()` pushing
  text to the native overlay via the Capacitor plugin bridge) simply
  never fires while backgrounded for more than a short window** — it's
  not that the calculation is wrong or that the native overlay fails to
  redraw; the JS timer driving the whole update never ticks. This holds
  regardless of what the interval callback computes, so the 2026-07-06
  coin-projection attempt (a display-only calculation invoked from the
  same interval) had no chance of working for the same reason the
  original refresh didn't — it was removed after on-device confirmation
  that it, too, doesn't update while backgrounded.

**Foreground Service option, evaluated and not pursued:** the only way to
keep something *reliably* ticking during this window is a genuine
Android foreground Service (a component type explicitly exempted from
this background throttling). Cost/risk considered:
- Requires the `FOREGROUND_SERVICE` permission plus, on Android 14+
  (API 34), a specific foreground-service-*type* permission (e.g.
  `FOREGROUND_SERVICE_DATA_SYNC` or `FOREGROUND_SERVICE_SPECIAL_USE`) —
  Google Play now scrutinizes foreground-service-type declarations, and
  "keep an idle-game simulation ticking for cosmetic overlay purposes" is
  not a clearly-sanctioned use case; risks review friction or rejection.
  and https://support.google.com/googleplay/android-developer/answer/13392821
  covers the current policy if this is revisited.
- Android mandates a **persistent, user-visible notification** for the
  entire time a foreground Service runs — a permanent notification-shade
  entry the player didn't explicitly ask for, for a small idle game.
- Meaningfully higher battery use, which Play Console's battery-usage
  reporting can surface against the app.
- The actual game simulation (`state`, `defeatEnemy()`, `damage()`, drop
  rolls, etc.) lives entirely in the WebView's JS context today. A
  foreground Service is plain Android/Java — it cannot directly execute
  that JS. Making the simulation *actually* progress from a Service would
  mean either (a) duplicating the core combat/loot rules in Java (a
  second implementation of the game's rules to keep in sync forever), or
  (b) some bridge that wakes the WebView briefly to run JS from the
  Service (adds its own complexity/reliability questions). This is a
  substantial new subsystem, not a small fix — consistent with the
  original 2026-07-05 (15) assessment.
- **Decision: not implemented.** The risk/cost (Play Store review
  exposure, a mandatory persistent notification, battery impact, and a
  real engineering lift to bridge native and JS simulation state) outweighs
  the benefit for a cosmetic overlay, especially given the safer
  alternative below covers the actual player need. Revisit only as a
  deliberate product decision, not as a follow-up bug fix.

**What shipped at the time (2026-07-06, Priority Round 3):**
`applyOfflineProgress()` (the deterministic catch-up that already runs on
return) also fires `showToast()`
(`おかえりなさい！留守中に${gain}cを獲得しました`) alongside its existing
`addLog()` line, so the moment the player actually returns, what they
earned while away is impossible to miss. At the time this was written, the
mini window itself was left as a frozen last-known snapshot while
backgrounded — see "Native live-ish projection" above for the Priority
Round 6 follow-up that added a native, non-WebView-JS coins/XP projection
on top of this (the return-time toast and guaranteed catch-up are
unchanged and still the source of truth).

**Correctness follow-up (2026-07-06, Priority Round 4):** the catch-up
formula itself was always correct, but its *input* wasn't reliable until
this round — `state.lastSeen` (the "when did the app last go away"
anchor) was being silently advanced by the routine `saveState()` autosave
(every 2.5s during normal play, and potentially a few more times early in
the background window before Chromium's throttling fully takes hold),
under-measuring how long the player had actually been away. Fixed by only
setting `state.lastSeen` at the two points that represent an actual
backgrounding transition (`visibilitychange`'s hidden branch,
`minimizeApp()`), never from the routine autosave. See DEV_LOG 2026-07-06
(19) for the full audit. Confirmed by design (not a bug): this catch-up
only ever grants a lump-sum coins+XP amount — it does not simulate chest
drops, rare-enemy spawns, or boss fights for elapsed offline time.

**Reliability follow-up (2026-07-06, Priority Round 5):** the Round 4 fix
alone still wasn't enough on a real device — the catch-up still didn't
reflect after ~1 minute backgrounded. Rather than keep tuning
`visibilitychange` in isolation, added `@capacitor/app`'s `pause`/`resume`
events as an independent, second trigger for `state.lastSeen`/
`applyOfflineProgress()` — these fire directly from Android's
`Activity.onPause()`/`onResume()`, not from the WebView's Page Visibility
API, so they don't share whatever failure mode (if any) affects
`visibilitychange` specifically. The existing `visibilitychange`-driven
BGM pause/resume and mini-window-close logic is untouched. Having two
independent trigger paths required making `applyOfflineProgress()`
idempotent (it resets `state.lastSeen` immediately after granting), so
neither path — nor both firing close together — can double-grant. See
DEV_LOG 2026-07-06 (20).

### Ad-open-chest button on the overlay (2026-07-06, Priority Round 6, item 2)

The mini window has a small "広告で\n開封" button next to the "×" close
control (`MiniWindowController`, same `TextView`-in-`LinearLayout`
pattern as the existing close button). It does **not** show an ad or open
a chest from the overlay itself — rewarded-ad SDKs render as a
full-screen Activity/Dialog and need an actual foregrounded Activity, not
an arbitrary `TYPE_APPLICATION_OVERLAY` window, so the button instead:
1. Sets a native, in-memory flag (`MiniWindowController.pendingAction =
   "openAdChest"`).
2. Calls the existing `openApp()` (identical to the double-tap gesture —
   launches the app, hides the overlay).
3. JS checks for this pending flag on *every* return-to-foreground path
   (cold start in `init()`, `visibilitychange`'s visible branch, the `App`
   plugin's `resume` listener — all three, for the same defense-in-depth
   reason `applyOfflineProgress()` is checked from all three) via a new
   `OverlayPermissionPlugin.consumePendingAction()` method, which reads and
   clears `MiniWindowController`'s flag so it only fires once per tap.
4. If the flag was set, JS calls the *existing*, unmodified
   `watchAdToOpenChest()` — same simulated-ad delay, same 5-minute
   cooldown check (silently no-ops if still on cooldown, matching the
   in-app button's existing behavior), same `grantChestLoot()` result.
   The mini window doesn't duplicate any of that logic — it only ever
   queues "please run the normal ad-open flow once you're back."

This keeps the mini-window button orthogonal to the real ad SDK question:
today, all ad flows in this game (both this one and the in-app "広告で
上限+10"/"広告で開封" buttons) are simulated placeholders (see
`PRIVACY.md`'s note on this), so there was nothing overlay-specific to
solve for a *real* SDK yet — when a real rewarded-ad SDK is integrated,
this button's behavior (queue the request, foreground the app, then run
the normal ad flow) shouldn't need to change, since it was already
designed around "the ad always runs in the foregrounded app," not from
the overlay.

### Correction: the game loop keeps running for real behind the mini window (2026-07-07, Priority Round 8)

Device testing revealed something that overturned a working assumption
from Priority Rounds 3–7: coins/XP were actually increasing for real
while the mini window was up, not just failing to *display* an increase.
The "Why: investigation findings" section above (Chromium throttles/
freezes WebView JS timers once hidden) turns out to describe
`requestAnimationFrame` specifically (which genuinely does stop once the
page is hidden — that part still holds and is why the native `Handler`
tick in "Native live-ish projection" above is still needed for the
*display*), but not the whole picture for this app: `loop()`'s own
per-frame combat logic (auto-attack, `hitEnemy()`, `defeatEnemy()`
chest/coin/XP rewards) was, in this project's specific configuration,
still executing somehow while backgrounded, genuinely advancing
`state.coins`/`state.xp` — most likely because Capacitor's default
`KeepRunning: true` (see "Why: investigation findings" above) keeps the
WebView's JS engine alive rather than pausing it outright, and this
Android WebView build evidently doesn't fully suspend the *`setInterval`/
game-loop-adjacent* timers the way a desktop-Chrome hidden tab would,
even though `requestAnimationFrame` callbacks specifically do stop.

This is good news for "feels like normal play while backgrounded," but it
exposed a real double-counting bug: `state.lastSeen` was only stamped at
the moment the app went hidden (`visibilitychange`, `minimizeApp()` —
see Priority Round 4's "Correctness follow-up" above), which treated the
*entire* background span as unaccounted-for "offline" time. Since the
loop was actually still granting real rewards during that span,
`applyOfflineProgress()` on return would then grant a *second*,
redundant catch-up on top of what was already earned for real.

**Fix:** `loop()` now stamps `state.lastSeen = Date.now()` on every frame
it actually executes real gameplay logic (gated on `state.started`, right
alongside the auto-attack/enemy-damage checks — see `app.js`). This does
not contradict Priority Round 4's fix (which was about not letting the
*routine, unconditional* `saveState()` autosave interval stamp
`lastSeen` regardless of whether real progress happened) — this new
heartbeat is tied specifically to the same code path that grants real
coins/XP, so it can only ever advance in lockstep with genuine progress,
by construction. Net effect:
- While the loop keeps genuinely running (foreground, or backgrounded
  behind the mini window, as confirmed on-device), `lastSeen` stays
  within a fraction of a second of "now" continuously, so
  `applyOfflineProgress()` correctly grants ~0 extra on return — no
  double-count.
- If the loop *does* genuinely stop (e.g. the OS eventually freezes the
  process — Priority Round 7's still-open App Freezer question),
  `lastSeen` stops advancing at exactly that moment, so
  `applyOfflineProgress()` still correctly grants catch-up for that
  genuine gap on return — the existing offline-reward safety net and
  formula are unchanged.
- Verified in the browser preview: simulating a hidden/visible cycle
  while the loop kept running granted zero extra catch-up; separately
  patching `lastSeen` to a genuinely stale value still correctly
  triggered a real catch-up grant. See DEV_LOG 2026-07-07 for the full
  test transcript.

**Also added, same round: a second, device-independent progress path.**
The `loop()` heartbeat above only helps if `requestAnimationFrame`'s
surrounding JS engine actually keeps `loop()` executing while
backgrounded — true in the browser preview test, but `requestAnimationFrame`
itself is specifically documented to stop once a page is hidden, and
whether the rest of the JS engine keeps ticking gameplay forward the same
way isn't something to bet an entire device's worth of behavior on. To
remove that dependency, the existing native 20-second `Handler` tick
(`MiniWindowController`, from "Native live-ish projection" above) now
*also* calls `WebView.evaluateJavascript()` to invoke a new
`window.__miniWindowTick()` (`app.js`'s `applyMiniWindowTick()`)
directly — since this is native Java injecting into the WebView, not a
JS-scheduled timer, it isn't subject to Chromium's background-tab timer
throttling at all, regardless of whether `loop()` itself is still
running.

`applyMiniWindowTick()` shares the exact same `grantElapsedProgress()`
helper (extracted from `applyOfflineProgress()`) and the exact same
`state.lastSeen` anchor as `applyOfflineProgress()` and the `loop()`
heartbeat — all three read and reset one shared timestamp, so whichever
of the three actually fires next simply resumes counting from wherever
the others left off. This makes a double-grant structurally impossible
between any pair of them, by construction, not by careful sequencing.
`OverlayPermissionPlugin.showMiniWindow()` hands `MiniWindowController` a
`WebView` reference (`getBridge().getWebView()`) for this; it's cleared
in `hide()` and the call fails silently (logged) if unavailable,
degrading to the pre-existing projection-only display rather than
crashing. Verified with a forced clean recompile
(`compileDebugJavaWithJavac --rerun-tasks`, confirmed it actually
re-executed rather than reporting a stale `UP-TO-DATE`) and
`assembleDebug`, both **BUILD SUCCESSFUL**; not yet verified on a real
device.

**Also fixed, same round:** the mini window's own ad-open-chest button
could get stuck showing "READY" after the ad had actually already been
opened and a fresh 5-minute cooldown set. Root cause: `hideMiniWindow()`
(called on every return-to-foreground, including the ad button's own
`openApp()` round-trip) resets `state.settings.miniWindowEnabled` to
`false`, and `updateMiniWindow()` used that same flag to decide whether
to push anything to the native side — so the fresh cooldown
`watchAdToOpenChest()` sets 900ms after granting never reached
`MiniWindowController`. Fixed by gating `updateMiniWindow()`'s native
push on a new `miniWindowEverUsed` flag (set once per session, on first
use, never auto-cleared) instead of `miniWindowEnabled` (which tracks
"is the overlay visible right now", a different question from "should we
still keep its cached state fresh"). The cooldown display itself (native
`MiniWindowController.updateAdButtonDisplay()`, added Priority Round 7)
was not changed — only ensuring the fresh deadline actually reaches it.

## Treasure Chests

Treasure chests are one of the most important reward moments.

- Base chest capacity is 10 total chests, counting normal and boss chests.
- If full, no more chests drop and the log says `宝箱がいっぱいです。`
- Reward ad grants chest capacity +10 for 3 hours. This is a capacity
  change only — it must not also open a chest. Opening chests (via
  "宝箱を開封") is a separate, unrelated action.
- Chest opening should be fast and dopamine-oriented.
- Special reveal animation is Epic or higher only.
- Higher rarity should use stronger but short effects.

### Legendary+ Full-Screen Reveal (2026-07-08, Priority Round 24)

Implements the design intent the user sketched earlier the same day (a
full-screen slot-machine-style reveal for Legendary+, reserved for those
rarities only so Common–Epic stays fast/tempo-focused). Built directly
against the live chest/rarity/gear/SE systems rather than as a standalone
mock, since the "does this actually feel good" judgment call needs the
real drop table, icons, and sound cues in place to be meaningful.

**Core principle: the roll is already final before any animation plays.**
`grantChestLoot()` still rolls the rarity, pushes the item to inventory,
updates `state.lastLoot*`, logs it, and calls `render()` exactly as
before — only *afterwards* does it branch on rarity to decide which
playback to show (`triggerLootReveal()`'s existing quick in-place flash
for Common–Epic, unchanged; the new `playChestReveal(item)` for
Legendary+). This means the reveal is pure presentation: skipping it at
any point has zero effect on what the player actually received, and a
debug replay can synthesize a fake item and drive the exact same
function real chest opens use.

**Screen structure** (`#revealOverlay` in index.html, z-index 90 — above
every other modal): a full-screen dimmed backdrop tinted by the item's
rarity color, a horizontal reel window (`#revealReelTrack`, plates
color-coded via `rarityColor()`/`rarityLabel()`), a chest icon
(`#revealChestWrap`) that shakes/bursts/wraps in a rainbow ring, a
particle layer for the burst, and a result card
(`#revealResult`) that fades/springs in with the final icon, name, and
rarity label. No new image assets — everything is the existing chest/gear
sprites plus CSS gradients, `conic-gradient` for the rainbow, and
JS-spawned spark divs.

**Reel mechanics**: `runReel()` builds one strip of `REVEAL_FILLER_COUNT`
(16) random low-rarity filler plates plus a single landing plate, then
animates the track's `translateX` (plain px math, not a `left:50%` CSS
offset — that combination was tried first and produced a real bug, see
below) over 1300ms with a `cubic-bezier` that looks fast-then-decelerate.
A handful of `setTimeout`-driven tick sounds (`playSound("chest")` — a
previously-unused audio asset, now correctly routed into the "loot"
volume group alongside drop/open/level) play during the scroll, spaced
with increasing gaps so the tick rate itself feels like it's slowing
down.

**Upgrade ("near miss") mechanic — the "never show higher then downgrade"
rule is structural, not a runtime check**: the reel only ever spins
*once* per reveal. Landing on a fake, one-tier-low rarity
(`REVEAL_FAKE_LOW_CHANCE`, 35%) and then "flipping" the same landing
plate up to the real stop (`flipPlate()`, a `rotateY` card-flip with the
content swapped at the midpoint) is the *only* mechanism for changing
what the plate shows, and every call site only ever flips upward:

- **Legendary**: reel lands on either Legendary directly, or (35%
  chance) fakes a landing on Epic first, then flips up to Legendary.
- **Mythic and above**: always plays the Legendary beat above first
  (with its own independent 35% Epic-fake chance) as stage 1. After a
  hold, stage 2 (`runSecondStage()`) wraps the chest in a spinning
  rainbow ring (`conic-gradient`) with a screen shake, then flips the
  *same* landing plate from Legendary up to the true (higher) rarity and
  bursts again, bigger (1.7x particle scale) — this doubles as the
  "looked like it landed on a lower prize, then upgraded" beat the user
  asked for, reusing the Legendary flow as stage 1 instead of a separate
  fake-out implementation.

Because every flip target is looked up as "one tier below the current
stop" or "the actual final rarity" (always ≥ what's currently shown),
there is no code path that can display a rarity and then replace it with
a lower one.

**Bug caught during verification**: the reel track's CSS originally kept
its old `left: 50%; transform: translate(-50%, -50%)` centering (from an

## Rare Enemies

Rare enemies are a growth and tap-check system.

- Rare enemies can appear in place of normal enemies.
- Rare enemies are visually highlighted.
- They have a time limit, shown to the player as a countdown badge and
  shrinking timer bar in the combat card.
- If not defeated in time, they escape.
- They have a higher chance to drop class shards.
- `huntRune` ("追跡のルーン") deals automatic periodic damage to the active
  rare enemy (once per second, scaled by rune level and the player's normal
  attack damage). Early game still favors tapping because the rune is
  expensive and its per-level output is modest; higher rune levels let
  well-invested players idle-clear rare enemies. Tune the per-tick scaling
  (`huntRuneDamage()` in app.js) after playtesting, same as Combat Power.

### Rare Kill Reward Boost (2026-07-08, placeholder numbers)

A rare kill previously granted the exact same reward roll as a normal
kill (shared with bosses: one 70%-chance class shard, one chest roll at
the normal drop rate) — killing one didn't feel meaningfully more
rewarding than a normal kill, just a bigger HP bar. Deliberately made
rare kills stand out:

- **Class shards:** 1 guaranteed shard per rare kill, plus a
  `RARE_ENEMY_CLASS_SHARD_BONUS_CHANCE` (currently `30%`) roll for a 2nd
  — instead of the shared boss/rare 70%-chance single roll. Originally
  shipped as a flat guaranteed 2, but the user found that made shard
  collection go too fast after reviewing it (2026-07-08) — switched to
  1 + a bonus roll so it's still clearly better than a normal kill
  without shortening the collection grind as much. Bosses keep their
  existing guaranteed single drop, unchanged.
- **Chests:** a bonus roll (`RARE_ENEMY_BONUS_CHEST_CHANCE`, currently
  `45%`) for a second chest, on top of the normal drop-rate roll — only
  checked if the first roll landed, so whiffing the base roll doesn't
  also grant a free bonus shot.
- Both existing chest-cap handling (`canReceiveChest()`/`chestLimit()`/
  `notifyChestFullIfNeeded()`) applies identically to the bonus roll —
  extracted into a shared `grantChestDrop(boss)` helper so there's one
  code path for "grant a chest, respect the cap, notify if full," not
  two.
- Both numbers are explicit placeholders, meant to be tuned after
  playtesting like the rest of the reward economy — the goal was just
  "clearly worth hunting down," not a final balance pass.

### 6-Slot Equipment (Implemented 2026-07-05)

Originally written up as a proposal (see DEV_LOG 2026-07-05 (13)); the
mechanical/data part was implemented in DEV_LOG 2026-07-05 (19) following
the "recommended approach" below, i.e. shipped without waiting on art.

**Current state (6 slots):** `slotLabels` in `app.js` = `blade`(武器),
`helmet`(兜), `armor`(鎧), `gloves`(手袋), `boots`(靴), `charm`(アクセサリ).

**Key constraint (still open):** `assets/game/manifest.json` has dedicated
pixel-art gear images only for `blade`/`charm`/`boots` (36 each).
`armor`/`helmet`/`gloves` — **3 of 6 slots** — have zero dedicated art and
render via the rarity-colored placeholder icon
(`applyGearIcon()`/`.icon-placeholder`, see DEV_LOG 2026-07-05 (5)). This
is a known, deliberate gap pending art production, not a bug.

**Balance change made alongside the slot expansion:** `gearPower()` and
`gearAffixTotals()` now scale their totals by `4 / (number of equipment
slots)` (`equipmentSlotScale()`), so equipping the 2 new slots doesn't
mechanically inflate Combat Power/defense/etc. by ~50% just from slot
count — the intent is roughly the same total-power ceiling as the old
4-slot system, redistributed across more items. **This ratio (4/6) is a
starting guess, not a playtested value** — needs the same playtest-based
tuning already called out for Combat Power elsewhere in this doc.

**Impact area covered during implementation:** `slotLabels`,
`gearNamePools` (helmet/gloves name pools added); `defaultGearSet()`
(already generic over `Object.keys(slotLabels)`, no change needed);
`migrateState()`'s `{ ...defaultGearSet(), ...state.gear }` merge (already

bump); `renderEquipment()`, `openChest()` slot selection, and
`fusionCandidates()`/`synthesizeSelectedGear()` (all already iterate
`Object.keys(slotLabels)` or filter by rarity only — slot-count-agnostic
by construction, no change needed). Equipment board CSS
(`grid-template-columns: repeat(3, ...)`) already lays out 6 items as two
full rows with no change needed.

## Normal Leveling via EXP (Implemented 2026-07-06, Option A)

User chose **Option A** (see options considered below): the coin
level-up path (`train()`, `levelUpCost()`, the "レベルアップ" button) has
been **removed entirely**. `state.level` now only ever increases through
combat/offline EXP (`state.xp` crossing `state.level * 7`, unchanged
formula) — there is exactly one leveling path, not two.

**What changed:**
- `train()`, `levelUpCost()`, and the `#upgradeButton` element/listener
  are gone. The status-screen stats list no longer shows a "レベルアップ"
  coin-cost line.
- `trainingRune` (`修練のルーン`) previously discounted `levelUpCost()`
  by up to 35% — that target no longer exists, so it was **repurposed**
  rather than deleted (deleting it would have silently voided any coin
  already invested by a player, with no in-game explanation). It now
  grants `経験値 +3%` per level, folded directly into `defeatEnemy()`'s
  `xpGain` formula alongside the existing `focus` skill and oracle
  mastery bonuses — same currency (coins), same rune slot, new effect
  that's still thematically "levels you up faster."
- Coins are now used only for gear/rune/ability costs (their pre-existing
  role) — no economy-wide rebalance of those costs was done alongside
  this change; flagged for playtest-based tuning like other balance
  numbers in this doc, since removing a coin sink changes how fast coins
  accumulate relative to what they're spent on.

Original proposal (kept for context on what was considered and why
Option A was picked over B/C):

Requested (device feedback): move normal (hero) leveling from coin
consumption to an EXP system, clearly separated in role from the existing
Job EXP system.

**Current state — important finding:** the game *already* has two
overlapping hero-leveling paths running at the same time, not a single
coin-only system to "migrate away from":

1. **Passive/combat EXP (already exists):** `state.xp` gains
   `xpGain = ceil((boss?3:1) * (1 + focus*0.1) * (1 + oracleMastery*0.04))`
   per kill (`defeatEnemy()`), plus a small amount from offline elapsed
   time (`state.xp += floor(elapsedMinutes / 2)`). When
   `state.xp >= state.level * 7`, it auto-levels: `state.level += 1`,
   `state.xp = 0`.
2. **Coin-purchased leveling (`train()` / the "レベルアップ" button):**
   spends `levelUpCost()` coins (`max(6, floor(level*16*(1 - up to 35%
   from trainingRune)))`) to do the exact same `state.level += 1`,
   independent of `state.xp`.

Both paths write to the same `state.level` with no coordination, so a
player can be leveling from combat *and* from spending coins
simultaneously — this is likely already a source of the "which currency
does what" confusion the device feedback is pointing at, not something
that only appears after a future change.

**Role separation (already true today, should stay true):**
- **Job EXP (`state.jobXp`, per class, via `gainJobXp()`):** drives class
  Job level and Job mastery bonuses (see "Classes And Jobs" above). Earned
  only from combat (`gainJobXp(boss ? 4 : 1)`), independent of which class
  is active having any bearing on `state.xp`.
- **Normal EXP/Level (`state.xp`/`state.level`, proposed to become the
  *only* leveling path):** should drive the hero's own combat stats
  (`damage()`, `heroHpMax()`, `hpRegen`, enemy scaling reference, etc.) —
  it already does this today via `state.level`, that doesn't change.

**Migration options considered:**
- **Option A — EXP-only, remove `train()`:** Delete the coin level-up
  button/cost entirely; leveling happens purely from combat EXP (already
  implemented) and possibly a larger passive-offline EXP rate. Coins get
  freed up entirely for gear/rune/ability costs (their existing role).
  Simplest mental model ("EXP levels you up, coins buy things"), but
  removes a coin sink, so coin balance across the whole economy
  (`abilityDefs` costs, `levelUpCost()` no longer competing for coins)
  needs a balance pass.
- **Option B — coins buy bonus EXP, not levels directly:** Repurpose the
  "レベルアップ" button to instantly grant a chunk of `state.xp` (e.g.
  equivalent to N kills' worth) for a coin cost, so leveling is *always*
  EXP-driven mechanically, but players can still spend coins to speed it
  up. Keeps a coin sink, keeps a single source of truth for what "level"
  means (state.xp crossing the threshold), least disruptive to existing
  formulas.
- **Option C — keep both, but decouple visually:** Leave both mechanisms
  as-is, just rename/re-explain the coin button so it's clearly framed as
  a separate "instant level" purchase rather than looking like part of the
  EXP bar. Lowest implementation cost, but doesn't actually resolve the
  "two systems doing the same thing" root cause.

**Recommendation at the time of writing:** Option B, for the reasons
below — preserved here for context since the user overrode it.

Option B preserves `levelUpCost()` as a coin sink (no economy-wide
rebalance needed), makes `state.xp` the single source of truth for
leveling (simpler mental model, and an EXP bar can be shown with one
meaning instead of two disconnected level-up buttons), and is a small,
mostly UI-and-one-function change (`train()` becomes "spend coins to add
XP" instead of "spend coins to add a level") rather than a deep formula
rewrite.

**Decision:** user picked **Option A** instead (simplest mental model,
accepted the coin-sink/economy-rebalance tradeoff) — implemented, see the
"Implemented 2026-07-06" summary above this proposal.
