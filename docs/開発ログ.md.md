### Root Cause Found: PluginCall.getDouble() Has No Long Branch (Priority Round 18)

With Round 17's logging fix in place, the user's next capture finally
showed the full JS-side chain working correctly end to end
(`[checkPendingMiniWindowAction] adChestRequested=true`,
`[watchAdToOpenChest] called`,
`[watchAdToOpenChest] setTimeout fired, new
adOpenCooldownUntil=1783474214530`, and the payload genuinely carrying
that value) — yet the very next native log showed it arriving as `0`.

Read `PluginCall.getDouble(String, Double)`
(`node_modules/@capacitor/android`) directly: it converts JSON values
that arrive as `Double`, `Float`, or `Integer` — no `Long` case exists.
`adCooldownUntil` is an epoch-ms timestamp (`1783474214530`), well
beyond Java's 32-bit `Integer` range, so Capacitor's JSON bridge encodes
it as a `Long`, and `getDouble()` silently fell through to its `0.0`
default. Every other field in the same payload (`coins`, `xp`, etc.)
stayed small enough to survive as `Integer`/`Double` for a fresh save —
this bug existed since Round 7 introduced the cooldown display, but
never triggered until a value this large needed to round-trip.

**Fix:** send `adCooldownUntil` as a string
(`String(state.adOpenCooldownUntil || 0)`), sidestepping the JSON-

Priority Round 4's fix: that fix targeted the *unconditional* autosave
interval (which could tick regardless of whether real progress
happened); this heartbeat is tied specifically to the same code path
that grants real rewards, so by construction it can only ever track
genuine progress. Net effect: `applyOfflineProgress()` now only ever
grants catch-up for a *genuine* gap where the loop truly stopped (e.g. an
eventual OS freeze — Priority Round 7's still-open question) — never for
a span that was already live-simulated. The offline-reward formula
itself is completely unchanged; only how `lastSeen` gets set changed.

### Native Handler-Based Mini-Window Coins/EXP Projection (Priority Round 6, item 1)

User asked to resume immediately after the Priority Round 6 handoff was
recorded — re-opened the mini-window-liveness question as top priority,
with the specific angle: can Android-native code alone, independent of
WebView JS, keep the overlay's numbers moving?

**Re-examined the Priority Round 3 conclusion instead of re-asserting
it.** That investigation correctly established that Chromium (Android
System WebView's engine) throttles/freezes JS timers running *inside the
WebView's own page* once backgrounded, regardless of Capacitor's Java-level
`keepRunning` setting — this is what killed both prior attempts
(2026-07-05's original refresh, 2026-07-06's `projectedOfflineCoins()`).
But that finding is specifically about the *WebView's JS execution
context* — it says nothing about the host Android app's *own* Java code
running on the main thread, which is a different execution environment
entirely. A `Handler.postDelayed` loop posted to the main `Looper` isn't
inside the WebView's Blink/V8 engine at all, so it isn't subject to that
specific throttling mechanism. This is the key distinction that makes a
native-only approach worth trying where JS-timer approaches weren't.

**Implementation:**
- Rewrote `MiniWindowController.java` to hold a small snapshot instead of
  a single opaque string: `baseTitle`, `baseStageLabel`, `baseHpText`,
  `baseChestText` (static display fields) plus `baseCoins`,
  `baseCoinsPerMinute`, `baseXp`, `baseXpPerMinute`, `baseXpNeeded`, and a
  `SystemClock.elapsedRealtime()` `baseTimestamp` marking when the
  snapshot was captured.
- New `updateProjection(...)` method replaces the old plain-text
  `update()` as the primary path (kept `update()` for backward
  compatibility with a bare `{text}` payload). Called from
  `OverlayPermissionPlugin.updateMiniWindow`, which now reads a `title`
  field to decide whether it received the new structured payload or the
  old plain-text one.
- `renderProjectedText()` computes
  `elapsedMinutes = min(240, (now - baseTimestamp) / 60000)` (same
  240-minute cap `applyOfflineProgress()` uses) and projects
  `coins = baseCoins + baseCoinsPerMinute * elapsedMinutes`,
  `xp = min(xpNeeded - 1, baseXp + baseXpPerMinute * elapsedMinutes)` —
  XP is deliberately capped just below the level-up threshold rather than
  simulating a level-up, since duplicating `state.level * 7` threshold
  logic natively would be a second copy of a game rule to keep in sync
  forever (same reasoning `docs/GAME_DESIGN.md` already applies to why a
  full foreground-Service simulation wasn't pursued, just scoped down to
  one specific rule instead of the whole combat system).
- A `Handler` (`android.os.Handler` on `Looper.getMainLooper()`) reruns
  `renderProjectedText()` every `TICK_INTERVAL_MS` = 20000 (20s, within
  the user's requested 10–60s range) via `postDelayed`. Started in
  `show()` (both the "create new overlay" and "already showing" paths),
  stopped in `hide()` via `stopTicking()` — no dangling timer once the
  overlay is actually closed.
- `app.js`'s `updateMiniWindow()` now sends a structured payload
  (`title`, `stageLabel`, `hpText`, `chestText`, `coins`,
  `coinsPerMinute`, `xp`, `xpPerMinute`, `xpNeeded`) instead of a single
  formatted string. `coinsPerMinute` is `damage() * 0.35` and
  `xpPerMinute` is a flat `0.5` — copied directly from
  `applyOfflineProgress()`'s real catch-up formula (not re-derived
  independently), so the native projection can't drift from what the
  real grant will eventually be.
- **No double-counting risk:** the native side is read-only with respect
  to game state — it only ever recomputes what to *display* from the
  snapshot; `state.coins`/`state.xp` are still only ever mutated once, by
  `applyOfflineProgress()` in JS, on actual return. The native projection
  and the real catch-up are two views of the same underlying rates, not
  two competing sources of truth.
- **No new Android permission, no persistent notification.** This is
  explicitly not the foreground-Service path Priority Round 3 evaluated
  and rejected — a plain `Handler` posting to an already-running app's
  main thread is a different, lower-overhead mechanism with a different
  risk profile. It still can't outlive the whole app *process* being
  killed by Android (e.g. low memory) — if that happens the tick stops
  along with everything else in the process, and the next cold start's
  `applyOfflineProgress()` call still settles the correct amount, same
  layered safety net as every other round this session.

## 2026-07-06 (18)

### Mini Window Realtime Update — Deep Investigation, Root Cause, Safe Fix (Priority Round 3)

User reported the Priority Round 2 item 3 coin-projection fix does not
actually work on device, and asked for a proper investigation rather than
another guess: what's happening to rAF/setInterval/visibilitychange while
minimized, why the display-only projection doesn't take effect, whether
Capacitor/Android has a safe fix, and a foreground-service cost/risk
writeup — explicitly no risky implementation.

**Investigated by reading Capacitor's actual Android source**, not just
re-reasoning from the existing docs:

- `android/app/src/main/java/com/onboardhero/game/MainActivity.java`
  extends `BridgeActivity` (no custom `onPause`/`onResume` overrides —
  everything goes through Capacitor's base implementation).
- `node_modules/@capacitor/android/.../BridgeActivity.java`: `onPause()`
  calls `bridge.onPause()`.
- `node_modules/@capacitor/android/.../Bridge.java`: `onPause()` calls
  `cordovaWebView.handlePause(keepRunning)`, where
  `keepRunning = shouldKeepRunning() || ...` and `shouldKeepRunning()` =
  `preferences.getBoolean("KeepRunning", true)`.
- Checked `android/app/src/main/res/xml/config.xml` (the legacy
  Cordova-style preferences file Capacitor's Android template still
  ships): no `<preference name="KeepRunning" .../>` entry present, so it
  falls back to the Java default, `true`.
- `node_modules/@capacitor/android/.../cordova/MockCordovaWebViewImpl.java`,
  `handlePause(boolean keepRunning)`:
  ```java
  if (!keepRunning) {
      // Pause JavaScript timers. This affects all webviews within the app!
      this.setPaused(true); // → webView.onPause(); webView.pauseTimers();
  }
  ```
  Since `keepRunning` is `true` for this app, this branch is **never**
  taken — Capacitor's own Java-level lifecycle handling does **not** call
  `WebView.pauseTimers()` for this project. At the Capacitor/Android-API
  level, `setInterval` should keep running while the Activity is paused.

**So why doesn't it work?** Android's System WebView is Chromium-based,
and Chromium has its own internal (Blink/V8 scheduler) background-page
power-management policy — intensified timer throttling that escalates
the longer a page stays hidden, and eventual full "freezing" of a hidden
page's JS execution after it's been hidden for more than a few minutes.
This operates *underneath* and *independently of* the Java-level
`WebView.pauseTimers()` API Capacitor controls — there's no Capacitor
config knob for it. This reconciles every observation:
- `document.hidden`/`visibilitychange` fire reliably on this device
  (confirmed working for BGM pause/resume and mini-window-close-on-return,
  Round 3/Round 6) — that's Chromium's own page-visibility *signal*,
  dispatched regardless of the throttling policy.
- `setInterval`-driven `updateMiniWindow()` calls stop being observed once
  the app has genuinely been backgrounded for a realistic duration
  (minutes, not seconds) — Chromium's own scheduler, not Capacitor,
  decides this, and it's a much harder stop than "throttled to ~1/min"
  once freezing kicks in.
- This is a **correction** to the assumption baked into the original
  mini-window fix (DEV_LOG 2026-07-05 (7): "`setInterval` keeps pushing
  the latest known state... unlike rAF") and the follow-up analysis
  (2026-07-05 (15)): true for short background windows, not for the
  realistic "minimized for a few minutes while doing something else"
  usage the mini window exists for.

**Consequence for the Round 2 item 3 fix:** the interval callback driving
`updateMiniWindow()` simply doesn't fire while backgrounded past a short
window — this is true regardless of what the callback computes. The
display-only `projectedOfflineCoins()` calculation added in Round 2 item
3 had no chance of ever showing up on-device, for the exact same reason
the base refresh doesn't. **Removed it** (`projectedOfflineCoins()` and
its call site in `updateMiniWindow()`) — it was dead weight giving a false
impression of a working feature; `updateMiniWindow()` now simply displays
`state.coins` directly again, matching what actually happens (a frozen
snapshot).

**Foreground Service cost/risk, as requested — evaluated, not
implemented:**
- Android 14+ (API 34) requires declaring a specific foreground-service
  *type* permission (e.g. `FOREGROUND_SERVICE_DATA_SYNC` or
  `FOREGROUND_SERVICE_SPECIAL_USE`), and Google Play now reviews these
  declarations for legitimate use — "keep a cosmetic overlay's numbers
  ticking" is not an obviously-sanctioned category, risking review
  friction or rejection.
- A foreground Service *mandates* a persistent, user-visible notification
  for its entire runtime — an always-on notification-shade entry the
  player never explicitly asked for, for a small idle game.
- Measurably higher battery consumption, which Play Console's own
  battery-usage reporting can surface as a black mark against the app.
- The real structural blocker: this game's entire simulation (`state`,
  `defeatEnemy()`, drop rolls, `damage()`, etc.) lives in the WebView's JS
  context. A Java Service cannot execute that JS directly. Making the
  simulation *actually* advance from a Service would require either (a)
  reimplementing the core combat/loot rules a second time in Java (two
  copies of the game's rules to keep in sync forever), or (b) some scheme
  where the Service periodically wakes the WebView to run a JS tick
  (adds its own reliability/complexity questions, and likely runs into
  the exact same Chromium freezing behavior for the WebView itself unless
  it's kept genuinely foregrounded, which is circular). This is a new
  subsystem, not a small fix — reconfirms the 2026-07-05 (15) assessment
  rather than overturning it.
- **Decision: not implemented**, per the user's explicit "don't force a
  risky implementation" instruction and this cost/risk balance.

**What ships instead — safe, no new permissions, no new subsystem:**
1. `docs/GAME_DESIGN.md`'s "Mini Window" section now states, as the
   *spec*, that the overlay does not support reliable real-time updates
   while backgrounded — a snapshot at minimize time, refreshed
   opportunistically at best, with a **guaranteed correct catch-up the
   moment the player returns**. This is now documented behavior, not an
   open bug to keep re-chasing with new `setInterval` tricks that can't
   work for the reason established above.
2. Strengthened the one channel that *is* reliable: `applyOfflineProgress()`
   (the deterministic return-time catch-up, unchanged formula) now also
   calls `showToast()` — `おかえりなさい！留守中に${gain}cを獲得しました` —
   alongside its existing `addLog()` line, so what was earned while away
   is impossible to miss right when it's actually delivered, rather than
   trying to preview it live in a channel Android's WebView won't
   reliably update.

**Verification:** `node --check app.js`. Live-tested the toast fix in a

**Found and implemented a safe, no-new-permission partial improvement:**
while the *game simulation* can't run in the background, the mini
window's `setInterval` tick *does* keep firing (throttled) even while
hidden — that's the entire reason `updateMiniWindow()` was moved off rAF

`projectedOfflineCoins()`:
```js
function projectedOfflineCoins() {
  if (!state.started) return state.coins;
  const elapsedMinutes = Math.min(240, Math.floor((Date.now() - (state.lastSeen || Date.now())) / 60000));
  return state.coins + Math.floor(elapsedMinutes * damage() * 0.35);
}
```
This is not a new formula — it's the *exact same* expression
`applyOfflineProgress()` already uses for the real catch-up grant on
return (same elapsed-minutes flooring, same 240-minute cap, same
`damage() * 0.35` rate). `updateMiniWindow()` now uses this projected
value for the displayed coin count only when `document.hidden` is true;
`state.coins` itself is never written by this — it's purely a display
computation layered on top of the real, unchanged value. This was a
deliberate design choice: because the display formula is identical to the
real catch-up formula, the number shown while backgrounded can never
drift from what `applyOfflineProgress()` will actually grant when the
player returns, and because `state.coins` is never mutated speculatively,
there is no double-counting risk when the real catch-up runs once on
return.

**Deliberately left un-projected:** HP, stage, level, and chest count in
the overlay text stay at their last real values. Coins are the only stat
the existing offline-progress model already simulates forward through
elapsed time (XP also technically accrues via `applyOfflineProgress()`,
but the mini window doesn't show XP/level-progress at all currently, and
adding a "projected level" display was judged out of scope for this
pass — the coin count is this game's headline "number going up" signal,
matching what the existing return-catchup log message already
emphasizes: "留守中に${gain}cを集めました。", not XP).

else.

**Caveat documented:** because the underlying interval is OS-throttled
while hidden (and Android's Doze/App Standby power-saving modes can
throttle it further under some conditions), the displayed number will
jump in coarser steps — as infrequently as once per elapsed real minute
— rather than a smooth per-second tick. This is a best-effort
approximation, explicitly not a guarantee, consistent with how this
project already frames the mini window as a "snapshot, not a live
simulation" (see `docs/GAME_DESIGN.md`).

## 2026-07-06 (14)

### EXP Leveling Bug Fixes + EXP Bar (Priority Round 2, item 1)

User reported EXP-based leveling (implemented last session, Priority
Round 1 item 3) "doesn't seem to be working," and asked for an EXP
progress bar. Investigated the actual gain/level-up code first rather
than assuming a specific cause.

**What was already correct:** `defeatEnemy()`'s `state.xp += xpGain` and
the `state.xp >= state.level * 7` level-up check were structurally sound,
and `damage()`/`heroHpMax()` already read `state.level`, so leveling
*does* affect combat stats once it happens.

**Bug 1 (the likely main cause) — no normalization for `state.xp`/
`state.level` in `migrateState()`.** Nearly every other state field in
this codebase gets a `typeof x !== "number"` guard in `migrateState()`
(pendingChests, rareEnemyUntil, bossIntroUntil, adOpenCooldownUntil,
etc.) — `state.xp`/`state.level` never did. `loadState()`'s
`{ ...createBaseState(), ...saved }` spread already protects against a
*missing* key (falls back to the default), but does nothing for a
*present-but-invalid* value — e.g. `NaN`, `null`, or a corrupted string.
If either field ever became invalid (save corruption, manual edits, or
any future code path that miscalculates before assigning), `state.xp >=
state.level * 7` would evaluate to `false` forever — the player would be
soft-locked at whatever level they were at, with no error, no log
message, nothing visibly wrong except levels never advancing again. This
matches "EXP leveling doesn't seem to work" precisely, and unlike a
subjective feel issue, is independently reproducible.

Confirmed with a live repro in a browser preview: wrote `xp: null,
level: "abc"` directly into `localStorage`'s save JSON, blocked the
autosave interval from immediately overwriting it (`localStorage.setItem
= () => {}` right before reload — otherwise the running `setInterval(saveState,
2500)` would silently re-save valid in-memory state over the corruption
before the reload actually took effect, masking the bug), then reloaded.
Without the fix, `heroHpMax()`/`damage()` would have received `NaN`
through `state.level`, corrupting HP/stat displays; with the fix
(`if (typeof state.level !== "number" || Number.isNaN(state.level) ||
state.level < 1) state.level = 1;` and the equivalent for `xp`, added
right after `state.saveVersion = SAVE_VERSION;` in `migrateState()`,
before anything else reads them), it correctly recovered to `level: 1,
xp: 0`.

**Bug 2 — level-up discarded overflow XP and capped at one level per
kill.** The check was a single `if`:
```js
if (state.xp >= state.level * 7) {
  state.xp = 0;
  state.level += 1;
  ...
}
```
Resetting to `0` instead of subtracting the threshold throws away any XP
earned past the exact threshold, and a single `if` (not a loop) means a
kill big enough to cross two or more levels' worth of XP (e.g. a boss
kill — `xpGain` is `boss ? 3 : 1` times the base — at a low level, where
`level * 7` is small) only ever grants one level. Changed to:
```js
while (state.xp >= state.level * 7) {
  state.xp -= state.level * 7;
  state.level += 1;
  ...
}
```
so multiple level-ups per kill work correctly and no XP is silently lost.

**EXP bar added:** `index.html`'s hero-panel "Lv" cell gained
`#expLabel` (`xp/needed` text) and `#expBar` inside a `.hp-track
.exp-track` div (reusing the existing HP-bar markup/CSS pattern, new
blue→purple gradient via `.exp-track span` in `styles.css` so it reads
as a distinct stat from HP). Widened the "Lv" grid column in
`.hero-panel` (`grid-template-columns`) since it previously only sized
for a bare level number. `render()` sets both from `state.xp`/
`state.level * 7`, same place `els.level.textContent` was already set.

**Verified live** in a browser preview (`python -m http.server`, new
`.claude/launch.json` added locally for this — not part of the app
itself): played through several kills, confirmed XP accumulates, the bar
fills proportionally (`2/14` → `14.2857%` width, matching exactly), a
level-up correctly bumped Lv 1→2 and HP max/attack increased immediately,
and the `trainingRune` "経験値 +N%" ability card (from Priority Round 1
item 3) still renders and its cost color-codes correctly against current
coins.

## 2026-07-06 (12)

### Combat Power / Boss Recommendation Structural Fix (Priority Round 1, item 2)

Before writing any numbers, asked the user which direction the mismatch
went (recommended power reached but boss still unbeatable, vs. recommended
power still too easy, vs. stage-dependent) — this is a subjective clear-feel
judgment only real play testing can answer, not something to guess at
blind. Answer: **reaching the recommended power still results in losing
badly to the boss.**

**Root cause found — this turned out to be a real structural bug, not
just a tuning nitpick.** `recommendedBossCombatPower()` was
`(area*88 + floor*24 + 120) * difficulty.scale`, a formula linear in
area/floor. The actual boss toughness the player fights
(`enemyHpMax()` when `isBossStage()`) compounds via `Math.pow(1.24,
area-1)` (exponential per area), a `bossScale = 4.4 + area*0.26` term
(itself growing with area), and the player's own current `state.level`
(bosses self-scale with player level) — none of which the linear
recommendation formula accounted for.

**Quantified the divergence with a standalone simulation** (same
formulas, run outside the game in plain Node) across areas 1–40 with a
rough level-progression assumption: the ratio of actual boss HP to the
displayed "recommended power" grows from ~1.4x at area 1 to over
**20,000x** by area 40. So "reach the recommended number, still lose
badly" isn't a subtle feel issue at all — past the earliest areas the
recommended number becomes almost meaningless, understating real boss
toughness by orders of magnitude. This directly explains the reported
symptom.

**Fix:**
- `enemyHpMax()` gained an optional `forceBoss` parameter
  (`forceBoss ?? isBossStage()`), so it can compute "what would this
  boss's HP be" without requiring the player to actually be mid-fight.
  The one existing caller (`resetEnemyHp()`) calls it with no arguments,
  so its behavior is completely unchanged.
- `recommendedBossCombatPower()` now computes
  `enemyHpMax(true) * BOSS_RECOMMEND_SCALE` instead of the old linear
  guess — i.e. it's now derived from the *actual* boss-HP formula (same
  area/floor/level compounding), not a separate approximation that can
  drift out of sync again by construction.
- `BOSS_RECOMMEND_SCALE = 0.7`: chosen so the displayed number stays
  close to the *old* formula's value at a representative first-boss
  scenario (area 1, floor 10, level 10 — where `448` was the old value
  and `enemyHpMax(true)` there is `637`, so `637 * 0.7 ≈ 446`). This
  keeps early-game numbers roughly where they were (no jarring change
  where there's been no complaint) while fixing the runaway divergence at
  higher areas — re-ran the same simulation with the new formula and
  confirmed the boss-HP-to-recommended ratio now stays constant
  (~1.43x) across all tested areas instead of diverging.

**Explicitly not claiming this is fully tuned:** the *shape* fix (tracking
the same compounding as real boss toughness) is code-derivable and
unambiguous given the measured divergence. The exact `0.7` scalar is a
reasonable starting calibration anchored to the old formula's early-game
value, not something verified against real playtest clear-rates —
flagged in `docs/TODO.md` with the same "needs playtest confirmation"
caveat already used for Combat Power/Job mastery/huntRune elsewhere in
this project.

Files touched: `app.js` only (`enemyHpMax`, `recommendedBossCombatPower`,
new `BOSS_RECOMMEND_SCALE` constant). Verified with `node --check app.js`
and a standalone Node simulation (not part of the shipped code) comparing
old vs. new formula output across areas 1–40.

setting `state.chestBoostUntil`, so a capacity increase correctly allows a

volumes/settings are untouched; only the `bgm` audio object is affected.

### Android Immersive/Fullscreen Mode (device feedback round 2, item 5)

**Where this belongs:** investigated first. No Capacitor status-bar plugin
(`@capacitor/status-bar` or similar) was present in `package.json`, and
this sandbox has no network access to add one via npm. Capacitor's own
config (`capacitor.config.json`) has no built-in "hide system UI" option
without such a plugin. So this was implemented **natively**, in
`android/app/src/main/java/com/onboardhero/game/MainActivity.java`, not
via Capacitor config.

**Implementation:** classic immersive-sticky flags on the decor view
(`View.SYSTEM_UI_FLAG_FULLSCREEN | ...HIDE_NAVIGATION | ...IMMERSIVE_STICKY`
etc.), applied in `onCreate()` and **re-applied in
`onWindowFocusChanged(true)`** — this second part matters: Android clears
these flags and brings back the status/nav bar whenever the window
re-gains focus (returning from the mini-window overlay, a system dialog,
the recent-apps switcher, or another app), so applying them only once in
`onCreate()` would make the immersive effect disappear after the very
first interruption.

**Deliberately not done:** did not add `androidx.core`
`WindowInsetsControllerCompat` (the modern, non-deprecated replacement for
`systemUiVisibility`) because it depends on the project's AndroidX version
being high enough, which could not be confirmed without a working Gradle
build in this sandbox. The classic flags are deprecated (API 30+
officially prefers `WindowInsetsController`) but still functional across
current Android versions, and don't require a dependency-version check
this sandbox can't perform. If Codex/the user confirm the AndroidX
version, swapping to `WindowInsetsControllerCompat` is a reasonable
follow-up for a cleaner, non-deprecated implementation.

**Real-device verification checklist (could not be done in this
sandbox — no Android SDK/Gradle, no device):**
- Confirm the build actually compiles (this sandbox cannot run Gradle).
- Devices with a display cutout/notch: check whether content is obscured
  or letterboxed around the cutout; may need
  `WindowManager.LayoutParams.layoutInDisplayCutoutMode` if it looks wrong.
- Gesture-navigation devices (Android 10+) vs. 3-button navigation: the
  "swipe from edge to reveal system UI" gesture behaves differently and
  can conflict with in-game edge swipes/taps; check the arena's
  `pointerdown` handling near screen edges doesn't fight with the OS
  gesture area.
- Older Android versions (below the `IMMERSIVE_STICKY` flag's
  practical floor) — confirm status bar hide still works acceptably,
  degrade gracefully if not.
- Confirm returning from the double-tap mini-window launch (which starts
  a fresh activity intent, see `MiniWindowController.openApp()`) still
  re-enters immersive mode correctly via `onWindowFocusChanged`.
- Different manufacturers' Android skins (Samsung One UI, etc.) sometimes
  override system-bar behavior; spot-check at least one non-stock-Android
  device if available.

## 2026-07-05 (15)

### Mini Window Background Updates (device feedback round 2, item 2)

**Investigation.** Two distinct problems were found, one platform-level and
one a real bug:

1. **Platform limitation (not fixable in JS alone):** the game's
   simulation (`loop()` — auto-attack, enemy attack, chest drops, etc.)
   runs on `requestAnimationFrame`. Android's WebView (like every modern
   browser engine) fully pauses `requestAnimationFrame` callbacks when the
   page is not visible (app backgrounded / minimized). This means the
   underlying game state stops changing the moment the app is minimized —
   there is no "live" coin/HP/stage progress happening in the background
   to report, independent of whether `updateMiniWindow()`'s own
   `setInterval` fires. `setInterval` timers are also throttled (though
   less aggressively) by the OS while hidden, compounding this.
2. **Real bug found and fixed:** `applyOfflineProgress()` — the existing
   lump-sum offline catch-up ("留守中に${gain}cを集めました。") — was only
   ever invoked once, inside `init()` at cold start. Returning from a
   minimized state within the same app session (`visibilitychange` firing
   `!document.hidden`) never re-ran it, so no catch-up coins/XP were ever
   granted for time spent minimized, and neither the main UI nor the mini
   window ever refreshed on return.

**Fixes applied:**
- `document.addEventListener("visibilitychange", ...)` now branches:
  - going hidden: `saveState()` (so `state.lastSeen` is accurate at the
    true backgrounding moment) + a final `updateMiniWindow(true)` push.
  - becoming visible: `refreshOverlayPermission()` (unchanged) +
    `applyOfflineProgress()` + `render()` + `updateMiniWindow(true)`, so
    returning from background now actually grants and displays the
    elapsed-time catch-up.
- `minimizeApp()` now calls `saveState()` and `await updateMiniWindow(true)`
  *before* triggering the native minimize, so the overlay shows an
  accurate snapshot at the exact moment of leaving instead of whatever the
  last `setInterval` tick happened to have (up to ~1s stale, or worse if
  mini-window mode was just enabled).
- Hardened `MiniWindowController.hide()` (Android,
  `android/app/src/main/java/com/onboardhero/game/MiniWindowController.java`)
  with try/finally around `windowManager.removeView()`, so an exception
  (e.g. the view already being detached) can no longer leave the overlay
  stuck open — state is always cleared regardless.

**Not fixed — needs a product decision, flagged for follow-up:**
True continuous live updates *while the app is backgrounded* (e.g. coins
ticking up second-by-second in the floating overlay while the user is in
another app) would require a **foreground service** to keep the process
un-throttled — this means a persistent Android notification, the
`FOREGROUND_SERVICE` permission (and a specific service type on newer
Android versions), and measurably higher battery use. This is a real
native-Android scope increase, not a JS fix, and changes the app's
resource footprint in a way that should be a deliberate product decision
rather than an incidental fix. Recommendation: treat the mini window as an
accurate **snapshot + on-return catch-up** tool (now fixed above) unless
the user/Codex specifically wants to invest in a foreground service for
true live background ticking.

**Verification:** `node --check app.js` passed. The Android/Java change
could not be compiled/run in this sandbox (no Android SDK/Gradle here);
needs a real build + device test to confirm `hide()` behavior and the
visibilitychange timing feel correct in practice.
